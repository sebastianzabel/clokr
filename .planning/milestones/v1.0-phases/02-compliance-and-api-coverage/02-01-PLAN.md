    Per D-03 and D-12, implement the missing 24-week rolling average in arbzg.ts and write tests.

    **Implementation in arbzg.ts:**
    Add a new check inside checkArbZG() (or equivalent function — read the file first to confirm the exact function signature and parameter structure). The check must:
    1. Query TimeEntry for the past 168 days (24 weeks x 7) for the employee where deletedAt IS NULL, type = 'WORK', endTime IS NOT NULL
    2. Sum net worked minutes across all entries (endTime - startTime - breakMinutes for each)
    3. Divide total net minutes by 168 (calendar days in the 24-week window) to get per-calendar-day average
    4. If average > 480 min (8h) per workday equivalent — specifically: total net minutes / number of Werktage (Mon-Sat) in the 168-day window > 480 min — push a warning with a NEW code (e.g., MAX_DAILY_AVG_EXCEEDED or similar) and severity "warning"
    5. Message in German: "Paragraph 3 ArbZG: 24-Wochen-Durchschnitt von 8 Stunden ueberschritten. Aktueller Schnitt: X.X h/Tag."

    **CRITICAL — averaging formula:** ArbZG §3 defines the 8h limit as per Werktag (Mon-Sat). The correct averaging denominator is the number of Werktage (Monday through Saturday) in the 168-day window, NOT the number of days the employee actually worked. A 24-week window contains exactly 144 Werktage (24 x 6). So the formula is: `totalNetMinutes / 144 > 480` triggers the warning.

    This means a 4-day/39h week (= 936h over 24 weeks) averages 936h / 144 Werktage = 6.5h per Werktag, which is < 8h and therefore LEGAL. A 5-day/45h week (= 1080h over 24 weeks) averages 1080h / 144 = 7.5h, still legal. A 5-day/48h+ consistent week would be 1152h / 144 = 8h exactly — borderline.

    Use date-fns-tz functions (already installed) for timezone-aware date computation. Use dateStrInTz() from timezone.ts to compute local calendar dates.

Create 24 weeks of synthetic TimeEntry records via app.prisma.timeEntry.create() in beforeAll of the describe block. Use dates in the 2025 range (past, stable). Clean up the separate employee in afterAll.

    Per D-01: if the implementation reveals any bugs, fix them inline and document in commit message.
    Per D-02: if a production bug is found, open a GitHub issue.
