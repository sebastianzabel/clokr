Test suite structure (one describe block per behavior domain):

**describe("POST /time-entries — locked month guard")**
- it: "returns 403 when target month has a SaldoSnapshot" — create snapshot for 2025-01, POST entry with date "2025-01-15", expect 403 with `{ error: "Monat ist abgeschlossen und kann nicht bearbeitet werden" }`
- it: "allows POST when target month has no SaldoSnapshot" — use a different month (2025-02), POST entry, expect NOT 403 (may get 201 or other validation error, but not lock error)
- it: "returns 403 even when no entries exist in the month yet (snapshot is authoritative)" — create snapshot for 2025-03 with NO time entries, POST entry to 2025-03, expect 403

**describe("POST /overtime/unlock-month — role check")**
- it: "returns 403 for EMPLOYEE role" — empToken, POST /unlock-month, expect 403
- it: "returns 200 for ADMIN role with valid snapshot" — adminToken, POST /unlock-month with snapshot existing, expect 200

**describe("POST /overtime/unlock-month — behavior")**
- it: "returns 404 when month has no snapshot" — adminToken, no snapshot for 2025-02, POST /unlock-month year:2025 month:2, expect 404 `{ error: "Monat ist nicht abgeschlossen" }`
- it: "deletes snapshot from DB after unlock" — create snapshot, call unlock, then query prisma.saldoSnapshot.findUnique, expect null
- it: "sets isLocked=false on all non-deleted time entries in that month" — create snapshot + create locked time entry (set isLocked:true manually in prisma), call unlock, fetch entry, expect isLocked=false and lockedAt=null
- it: "does not unlock entries from a different month" — create snapshot for Jan, create locked entry in Feb, call unlock for Jan, verify Feb entry still has isLocked=true

**describe("POST /overtime/unlock-month — tenant isolation")**
- it: "returns 404 when employeeId belongs to a different tenant" — seed a second tenant via prisma.tenant.create (not using seedTestData to avoid complexity), create employee in second tenant, try to unlock with adminToken from first tenant, expect 404

**describe("POST /overtime/close-month — earlyClose response")**
- it: "includes earlyClose:true when called before day 15 of the following month" — this test is timing-dependent. Since tests run on 2026-04-13, close a month that ended recently (March 2026, if before April 15). Strategy: determine current date in test; if it's before the 15th of this month, close the previous month and expect earlyClose:true; otherwise skip with a comment. Alternatively: ALWAYS close a future month (2099-01) which is before any day-15 — but that's blocked by the "no future months" check. Better: use a real historical month (2025-01) and verify earlyClose is FALSE (the grace period for Jan 2025 ends Feb 15 2025, which is in the past). Flip test: close January 2025, verify earlyClose is false (grace period has long passed). For the earlyClose:true case, note this is hard to test deterministically without date-mocking; test the false case instead and document in a comment that earlyClose:true requires calling before day 15 of the following month.
