      // § 5 Abs. 2 BUrlG: H1 exits are capped at pro-rata entitlement

        select: { exitDate: true },

      const exitDate = empForExit?.exitDate ?? null;
      if (
        exitDate &&
        exitDate.getFullYear() === year1 &&
        exitDate.getMonth() < 6

        const used = Number(ent1.usedDays);
        if (split.year1Days > proRata - used) {
          return reply.code(400).send({
            error: `Anteiliger Urlaub bei Austritt in H1 überschritten (${proRata} Tage anteilig)`,
            available: proRata - used,
            requested: split.year1Days,
          });
        }
      } else if (split.year1Days > avail1) {

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-02-01 | Elevation of Privilege | Booking H1 cap: employee bypasses pro-rata cap | mitigate | Cap check reads exitDate from DB scoped by `tenantId`; client cannot supply exitDate. Any mismatch between client-supplied dates and DB state is caught by the entitlement check. |
| T-16-02-02 | Spoofing | employeeId in /entitlements path param | mitigate | Handler already uses `tenantId` from JWT in all DB queries; cross-tenant access is structurally prevented by `where: { id: employeeId, tenantId }` |
| T-16-02-03 | Information Disclosure | effectiveEntitlementDays exposes HR data | accept | Endpoint already requires auth (`requireAuth` preHandler) and returns the employee's own entitlement data; no new sensitive fields beyond what totalDays already discloses |
| T-16-02-04 | Denial of Service | extra DB query per booking/entitlement call | accept | Single `findUnique` per call; indexed by primary key; negligible overhead |
</threat_model>
