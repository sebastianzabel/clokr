Isolate test runs to a separate PostgreSQL schema (`test`) so tests never write to the dev database.

Purpose: Currently all tests share the dev database, preventing parallel execution and risking data corruption. Per D-01/D-02: use same PG instance with `?schema=test` parameter. Per D-03: truncate-per-suite (not transaction rollback) is the correct cleanup approach for Prisma with connection pooling.
Output: `.env.test` with TEST_DATABASE_URL; `vitest.setup.ts` overrides DATABASE_URL before app boots; `pretest` pushes schema to test DB; `setup.ts` cleanup wrapped in try/finally.
</objective>

    ```typescript
    import { config } from "dotenv";
    import { resolve } from "path";

    // globalSetup runs before any test worker — override DATABASE_URL here
    // so the Fastify app connects to the test schema, not the dev database.
    export function setup() {
      config({ path: resolve(__dirname, ".env.test"), override: true });
      if (!process.env.TEST_DATABASE_URL) {
        throw new Error(
          "TEST_DATABASE_URL is not set. Create apps/api/.env.test with TEST_DATABASE_URL pointing to ?schema=test"
        );
      }
      process.env.DATABASE_URL = process.env.TEST_DATABASE_URL;
    }
    ```

    Note: `globalSetup` modules must export a `setup()` function (named export), not run side effects at module level. The `override: true` flag ensures `.env.test` values win over any already-loaded `.env` values.
